What this is

  
A schema-driven playground for search APIs across providers, with two modes:

  
    
    
  • Playground — each provider is a tab (Brave · Exa · Parallel · Perplexity · SerpApi · Tavily · You.com); pick an endpoint, fill a form generated from its schema, and inspect the raw response.
    • 
    
  • Compare — configure up to four comparable endpoints (provider + endpoint + parameters), ask one question, and see each endpoint's raw response side by side — the same view as the provider tabs.
    • 
  


  


  
Design principle — one provider, one set of files

  
The goal: if someone wants to work on Perplexity, its code is one clear package, not logic strewn across the server and the frontend. So every provider is a self-contained package that implements a shared contract, the server is a thin dispatcher over a registry, and the frontend is a generic renderer with no provider-specific code.


  
competitor_search/
├── providers/            # one package per provider — the core of the app
│   ├── base.py          # the contract: Param · Endpoint · Provider (+ compare)
│   ├── registry.py      # the one list of which providers exist
│   ├── exa/             # every provider/ folder has the same 3 files (below) — Exa shown as the example
│   │   ├── __init__.py  # exports the Provider subclass
│   │   ├── provider.py  # ExaProvider: identity, auth, endpoints, call()
│   │   └── endpoints.py # /search /contents /answer schemas
│   ├── youdotcom/       # You.com — search / contents / research / finance_research
│   ├── perplexity/      # Perplexity Agent API (Bearer auth, streaming call())
│   ├── parallel/        # /v1/search (mode: turbo/basic/advanced) + /v1beta/search + /v1/tasks/runs (Task API) + FindAll
│   ├── tavily/          # Tavily — search / extract / map / crawl (Bearer)
│   ├── brave/           # Brave — GET query-param API (web/news/images/videos + suggest/spellcheck/summarizer/local)
│   └── serpapi/         # SerpApi — GET /search across engines (google/bing/ddg/…); ?api_key= query auth
├── app/                 # thin web layer over providers/
│   ├── server.py        # /api/providers · /api/call · /api/compare
│   ├── playground.js    # generic renderer — fetches catalog, no per-provider code
│   └── index.html       # tabs, form shell, compare picker, this About page
└── comparison/          # perplexity_runner.py — SSE Agent-API runner used by call()


  


  
The core classes providers/base.py

  
Three classes are the whole contract. A new provider is usually just declarative — no imperative code.

  

    
ClassIsKey parts

    
      
ParamOne request-body fieldname, type (string·text·int·bool·enum·csv·date·json·group), values, nested fields for objects. Flags steer the form: required, optional (group enable), deprecated (hidden behind a toggle), advanced (tucked into a collapsed Advanced options group). Renders to a form control; serialized via to_dict().

      
EndpointOne API operationmethod, path, and its list of Params.

      
ProviderA providerbase_url, auth_header/auth_prefix, key_env, endpoints. call() runs the request behind both the playground and Compare (default = plain JSON POST; override for streaming/async); catalog() serializes it for the UI.

    
  

  
Providers differ — Exa is a plain JSON POST with x-api-key; Perplexity's agent is SSE-streaming with Bearer auth. The base handles the common case; a provider that needs streaming or custom auth overrides call() inside its own package (Perplexity does). Consistency from the base, isolation per provider.

The request-body schema has a single source of truth — the provider's Python package. The frontend never hardcodes it; it fetches the catalog and renders. So schema and routing can't drift apart. Any endpoint with a compare_query_field is automatically selectable in the Compare picker.

  
The generated form keeps a clean primary view: short fields pair up two-per-row, roomy fields span the row, help lives in a hover ⓘ tooltip, booleans are toggle switches, and less-common params collapse into Advanced options.


  


  
Adding a provider

  
    
    
  1. Create providers/<name>/ with a Provider subclass and its endpoint schemas.
    2. 
    
  2. Register it in providers/registry.py.
    3. 
    
  3. Add its API key to env.txt.
    4. 
  

  
It then appears everywhere — a playground tab, its endpoint dropdown, and generated request forms — with no changes to server.py or playground.js. Any endpoint that sets compare_query_field is automatically selectable in the Compare picker. Override Provider.call() only if the API streams, is async, or authenticates unusually.
